诸神愚戏副本论坛网站下一步功能计划表

更新时间：2026-07-12


一、当前判断

网站已经完成副本档案、评分证言、通关登记、个人档案、榜单、结算信封、称号诅咒、天赋仓库、微信召集和基础安全加固。

当前最需要补的不是继续堆新玩法，而是把后台状态看清楚、把管理动作做成闭环、把生产异常处理从“查 SQL”变成“馆主后台可见、可审计、可修复”。

已完成的前置稳定性改动：

1. 分数结算已增加重复提交防护和幂等请求 ID。
2. 昵称已改为身份绑定字段，只有馆主可以更改。
3. 非馆主登录后隐藏昵称绑定入口，避免误以为可以改名。
4. 常见写操作已增加前端互斥锁，降低重复点击造成重复写入的风险。
5. 个人页和楼主提醒已改为优先按邀请码 hash 查询自己的构筑记录，不再只靠昵称匹配。


二、总体方向

下一阶段建议按四层推进：

1. 馆主后台一期：玩家查询、称号/诅咒管理、天赋异常扫描与修复。
2. 操作日志：统一记录后台管理动作，形成可追溯审计链。
3. 玩家成长时间线：把通关、结算、称号、诅咒、天赋变化串成履历。
4. 活动/赛季与运营工具：让副本、榜单、召集和作者反馈支持阶段性运营。

推荐原则：

- 先做后台可见性，再做新玩法。
- 先做只读查询，再做写入修复。
- 所有敏感写操作继续走 Supabase Edge Function。
- 前端隐藏只做体验，真正权限以后端为准。
- 每次涉及 SQL 或 Edge Function，都按“SQL 剪贴板 -> Edge Function 剪贴板 -> GitHub 推送/确认”的方式分批交付。

四、第一批建议开工范围

第一批只做“馆主后台一期”，不要一次性扩展太多。

推荐实现范围：

1. 新增 admin 可见的“馆主后台”入口。
2. Edge Function 新增 `adminLookupPlayer`。
3. 后端返回：
   - 玩家档案
   - 邀请码角色
   - 登神之路 / 觐见之梯
   - 称号列表
   - 诅咒列表
   - 天赋仓库
   - 携带槽
   - 碎片数量
   - 溢出项
   - 最近结算信封
4. 前端新增玩家查询面板。
5. 面板内支持：
   - 回收指定称号
   - 回收指定诅咒
   - 修复天赋异常
6. 所有动作完成后立即刷新查询结果。
7. 普通玩家、作者、审核员、神明不能打开完整馆主后台。


五、第一批验收标准

1. admin 能打开馆主后台。
2. 非 admin 访问后台入口会被前端隐藏，直接调用后端也会被拒绝。
3. 输入玩家昵称能查到完整玩家状态。
4. 能看到该玩家所有 active/非 active 称号。
5. 能从后台回收指定称号并立即看到状态变化。
6. 能看到该玩家所有 active/非 active 诅咒。
7. 能从后台回收指定诅咒并立即看到状态变化。
8. 能看到该玩家天赋仓库、携带槽、碎片、溢出项。
9. 能执行一次天赋异常扫描。
10. 能执行一次可自动修复的天赋异常修复并刷新结果。
11. 后端不向前端返回邀请码明文或可复用的敏感 hash。
12. `index.html` 内联脚本解析通过，`git diff --check` 通过。


六、第二批建议开工范围

第二批做“操作日志一期”，给后台动作留痕。

推荐实现范围：

1. 新增 SQL 表 `admin_operation_logs`。
2. 新增统一后端日志写入 helper。
3. 称号下放、称号回收、称号恢复写日志。
4. 诅咒下放、诅咒回收、诅咒恢复写日志。
5. 天赋异常扫描和修复写日志。
6. 结算撤销写入统一后台日志视图。
7. 馆主后台显示最近 50 条管理操作。


七、第三批建议开工范围

第三批做“玩家成长时间线”，先后台完整，再公开精简。

推荐实现范围：

1. 后台整合玩家通关记录。
2. 后台整合玩家分数结算与撤销记录。
3. 后台整合称号、诅咒变更记录。
4. 后台整合天赋抽取、兑换、装备、分解记录。
5. 公开档案展示精简时间线，只展示适合公开的信息。
6. 时间线支持按类型筛选。


八、为什么先做馆主后台

最近出现的问题主要集中在后台状态不可见：

- 回收称号后，不确定是否真的回收成功。
- 仓库还有空位，但前端显示溢出。
- 天赋携带规则被隐藏逻辑挡住。
- 结算重复提交需要从 Supabase SQL Editor 查。
- 昵称作为身份字段，如果没有后台统一入口，很容易被误认为玩家自己可改。
- 管理动作成功与否缺少统一审计记录。

因此下一步最重要的不是新玩法，而是让馆主能直接看到和修复玩家状态。


九、后续开发注意事项

1. 后台管理功能必须只对 admin 开放。
2. 神明账号可以保留称号/诅咒相关能力，但不能进入完整馆主后台。
3. 玩家、作者、审核员都不能更改昵称。
4. 所有后台写操作都必须走 Edge Function，不直接从前端写表。
5. 查询接口可以返回后台需要的信息，但不要向普通玩家暴露邀请码哈希。
6. 不要提交 `.codex/`、`private_invites/` 或任何明文邀请码。
7. 不要运行 `second_beta_reset.sql`，除非明确要重置二测数据。
8. 每次涉及 Supabase Edge Function 修改，GitHub 推送后仍需要用户手动部署 Edge Function。
9. 每次涉及 SQL 新表或索引，必须把 SQL 内容复制到剪贴板给用户执行。
10. 每批交付后都要明确说明：GitHub 是否已推送、SQL 是否已执行、Edge Function 是否已部署。


十、当前推荐下一步任务

下一次正式开工建议任务名：

“实现馆主后台一期：玩家查询、称号/诅咒管理、天赋异常修复”

建议交付批次：

1. 第一批：SQL 表/索引/辅助函数，复制到剪贴板，由用户在 Supabase 执行。
2. 第二批：Edge Function 新增后台查询和后台写操作，复制到剪贴板，由用户部署。
3. 第三批：GitHub 前端推送，包含馆主后台入口和查询面板。
4. 第四批：线上烟测，确认 admin 可查、非 admin 被拒、回收/修复动作可刷新。
